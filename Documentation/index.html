Read Me

Based on feedback from Project 1 and conversation with Craig, I tried to prioritize the user experience on this project. I started with looking at how people would access this project in my target sites i.e. South Asia mainly, and noted that most online experiences would be through mobile phones. Thus, my design was mobile first.

Focus on one user experience

Though this project was conceputalized as a larger platfrom "Digital Futures Initiative" that will accomodate many kinds of content that is useful in the global south. I did not choose to start by building out the platform. Rather I took one small piece of that overall experience: a specifc lesson on image manipulation that will fall under the digital citizenship content and built that out.

I found that process very useful because it actually began to reveal what would be needed from one users prespective, which I think enriched my understanding of the system. For e.g it was not clear to me initially if a chat/socket type solution would fit into this project. But as I built the user experience out, it has become clear that such a technical implementation would be useful each time a facilitator tries to run a session.

Technical Resources

Technically the project uses Nodejs and Express to spin up a server and serve both html and js pages up dynamically. It also uses Pug as a templating engine. For the Nodejs and Express I largely found the resources shared in class to be sufficient with some use of stackoverflow/stackexchange to support problems.

For Pug, I relied heavily on Pug documentation and also stackexchange.

Technically the hardest part was to conceptualize the app. Since I have not really worked with webtech before I was not sure if doing it the way I did (putting all logic on the server) was appropriate or if more the logic should be pushed client side.

To help understand this, I did about 3 "build your own app with MongoDB and Node" tutorials from youtube. These helped me see how others think about the project architecture. BUT it feels like the tutorial projects are way too simple to actually inform my choice. Because I was feeling stuck, I decide to just pick a path. I picked the templating engine/server side route mostly because I had never done it before and it was a good way to learn.

For the rest of this semester, I am committed to this architecture choice, but as I plan to expand this project into my final project. I plan to revisit it. I am curious if just sending a JSON to the client, serving it up a js file and letting it do all the processing makes sense. Or if such a choice would actually make it easier to work in low bandwidth environments.

Next Steps
